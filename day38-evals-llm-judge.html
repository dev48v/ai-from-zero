⚖️ Evals & LLM-as-a-Judge — a benchmark gives one number for a leaderboard; a task eval tells you it works for you. And a model can grade another model — if you cancel the biases: position, verbosity, self-preference

You built a smart model. Now — is it actually good? And can another model be the judge?

Human grading is the gold standard, but it's slow and expensive. So we ask a strong model to grade — an LLM-as-a-judge. It agrees with human preference about as often as two humans agree with each other, at API speed and near-zero cost. The catch: the judge inherits biases a tick-box test never had. It quietly prefers the answer shown first (position), the longer answer (verbosity), and text from its own model family (self-preference). Below, Answer A is genuinely the better answer — flip a bias and watch the judge change its mind for the wrong reason, then switch on a mitigation and watch it recover. Everything runs in your browser, no model calls.

② Pointwise eval suite — N real tasks → a rubric score each → one aggregate you can trust

The other half of evals: not "A vs B" but "does the answer clear the bar?" Each case gets a 0–5 rubric score from the judge; a case passes if it's ≥ the pass bar. The aggregate is reproducible — the same inputs give the same number every run, so it catches regressions a vibe-check would miss.

Both halves matter. Pairwise tells you which of two models people prefer (with the order-, length- and family-biases carefully cancelled); pointwise gives a stable, reproducible score you can gate a release on. Neither replaces a small human-labelled anchor set that you keep the judge honest against.

🔨 Build a de-biased LLM-judge harness

    
An eval you can gate a release on is more than "ask a model which is better." Here it is in annotated pseudo-code — a reproducible pointwise suite, a rubric-anchored judge prompt, a pairwise call, then the three de-biasing moves (swap-and-average, rubric/length control, blind + diverse panel), and finally calibrating the judge against humans.

    
    
      
  1. 
        
    1
    The pointwise eval loop — a reproducible score
    
        
    Start with your tasks, not a public benchmark. Run each case, score the answer against a rubric, and pass if it clears a bar. Because it's deterministic, the same inputs give the same number every run — so a drop is a real regression, not noise.
    
        
    def run_eval(model, cases, judge, pass_bar=3.0):
    results = []
    for c in cases:                                  # YOUR real inputs
        answer = model(c.input)
        score  = judge_pointwise(judge, c.input, answer, c.rubric)  # 0..5
        results.append(score >= pass_bar)            # PASS / FAIL
    return mean(results)                             # one aggregate number
# same inputs -> same number. a drop between runs = a real regression.
    
      
    2. 
      
  2. 
        
    2
    Write the judge prompt — a rubric + a parseable verdict
    
        
    A vague "which is better?" invites every bias. Anchor the judge on explicit criteria, tell it to ignore length and confidence, and force a structured output you can parse. A rubric + a fixed schema turns a vibe into a number.
    
        
    JUDGE_PROMPT = """
You are grading an answer. Score 0-5 on THESE criteria ONLY:
  - correctness  (is it factually right?)        # anchor on substance,
  - relevance    (does it answer the question?)  # NOT length or style
  - instructions (did it follow the format?)
Ignore how long or confident the answer sounds.
Question: {q}
Answer:   {a}
Reply as JSON: {"reasoning": "...", "score": <0-5>}   # force a parseable verdict
"""
    
      
    3. 
      
  3. 
        
    3
    Pairwise — comparing is more reliable than absolute scoring
    
        
    Judges (and humans) are far more consistent at comparing two answers than at giving an absolute 0–5. So for head-to-heads, show both answers and ask which wins. But this opens the door to the one bias absolute scoring doesn't have: the order of [1] and [2].
    
        
    def judge_pairwise(judge, q, ans1, ans2):
    prompt = f"Question: {q}\n[1] {ans1}\n[2] {ans2}\n" \
             "Which answer is better? Reply 1 or 2. Judge on the rubric, not length."
    return judge(prompt)          # -> "1" or "2"
# more reliable than absolute scores... but sensitive to the ORDER of [1]/[2].
    
      
    4. 
      
  4. 
        
    4
    Swap-and-average — kill position bias
    
        
    Many judges systematically favour whichever answer sits in position 1. So judge both orders and only count a win if the same answer wins both ways. The order-dependent bonus lands on slot 1 in each run, so it cancels — exactly the "🔁 Swap & average" toggle you flipped in LOOK.
    
        
    def debiased_pairwise(judge, q, a, b):
    first  = judge_pairwise(judge, q, a, b)   # A in slot 1
    second = judge_pairwise(judge, q, b, a)   # B in slot 1  (swapped!)
    if first == "1" and second == "2":  return "A"    # A won BOTH orders
    if first == "2" and second == "1":  return "B"    # B won both orders
    return "TIE"                                        # order flipped it -> tie
# the slot-1 bonus appears in both runs, so it CANCELS out of the comparison.
    
      
    5. 
      
  5. 
        
    5
    Rubric + blind + panel — kill verbosity & self-preference
    
        
    Swap-and-average fixes order, not length or family. For verbosity, anchor on the rubric and tell the judge not to reward length. For self-preference, keep the judge blind to who wrote each answer and average a panel of different-family judges.
    
        
    # verbosity: anchor on a rubric, forbid rewarding length
prompt += "Do NOT reward longer answers. Judge correctness & relevance only."

# self-preference: blind the judge + use a PANEL of different families
answers = shuffle_and_strip_model_names([a, b])   # judge can't see who wrote what
judges  = [judge_vega, judge_nova, judge_atlas]    # different model families
verdict = majority(debiased_pairwise(J, q, a, b) for J in judges)
# never let a model be the sole judge of its own contest.

    Calibrate against humans, then gate the release
    
        
    Is the judge trustworthy? Compare it to a small human-labelled anchor set and measure agreement (and Cohen's κ, which corrects for lucky agreement). Only once the judge tracks humans do you let it block a release when quality regresses.
    
        
    # is the judge trustworthy? check it against HUMAN labels.
agree = mean(judge_label[i] == human_label[i] for i in anchor_set)
kappa = cohens_kappa(judge_labels, human_labels)   # chance-corrected agreement
assert agree > 0.8 and kappa > 0.6, "judge not reliable enough to gate on"

# only NOW gate the release on the eval suite:
if run_eval(candidate) < run_eval(current) - 0.02:
    block_release("quality regressed on the eval suite")

🎉 Day 38 of AIFromZero done.
You saw how we actually measure "good": benchmarks give a comparable leaderboard number, task evals tell you it works for your job. You can grade with pointwise rubric scores (reproducible, great for regressions) or pairwise A-vs-B (more consistent, but order-sensitive). And a model can be the judge — if you cancel its biases: swap-and-average for position, rubric anchoring + length control for verbosity, and blind + diverse panels for self-preference — all kept honest by a human anchor set. 👉 Tomorrow — Day 39: Synthetic Data & the Self-Improvement Loop — once you can measure quality cheaply, you can generate training data and let a model filter its own outputs to get better.
