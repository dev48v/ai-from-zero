AIFromZero · Day 39

♻️ Synthetic Data & the Self-Improvement Loop — once you can cheaply verify quality, a model can generate its own training data, filter it, and retrain to get better each round. Skip the filter and it eats its own tail: model collapse.

You ran out of human data. So the model makes its own — then grades and retrains on it. Does that actually work?

The trick is an asymmetry: for many tasks verifying an answer is far cheaper than generating one (checking a proof, running unit tests, matching a known answer). So a mediocre model can spray out many candidates, a cheap verifier keeps only the good ones, and the model retrains on that cleaner-than-average set — a bit better each round. That's STaR / self-training / distillation. But the filter is the whole game: retrain on unfiltered output and quality stalls — worse, if a model learns generation after generation only from its own samples, the distribution narrows until it collapses to a single note. Both demos below run entirely in your browser, no model calls, fully deterministic.

① The self-improvement loop — generate candidates → verify/filter → retrain → repeat, and watch quality tick up

Each round the model generates 12 candidate answers; a verifier keeps the ones it can confirm are correct; the next generation trains on that kept set. The next model's quality moves toward the purity of the kept set — so a good filter lifts it, and no filter goes nowhere. Drag the filter to 0% and the loop flat-lines: garbage in, garbage out.

② Model collapse — train a model on its own unfiltered output, generation after generation

Here there's no verifier — each generation simply learns from samples of the previous one. Sampling over-weights the peak and drops the rare tails, so the distribution keeps narrowing. With 0% real data it collapses to a single spike (diversity gone). Slide in some real/human data and it re-anchors the tails and stays healthy.

🔨 Build a self-improvement (STaR) loop — with a collapse guard

    
Turn a model plus a verifier into a data factory. Generate many candidates, keep only the ones a verifier confirms, fine-tune on the kept set, and repeat — the loop from LOOK ①. Then the distillation variant (a strong teacher makes data for a small student) and the guardrails that stop LOOK ② — mixing in real data and watching diversity so the model never eats its own tail.

    
    
      
  1. 
        
    1
    Generate candidates — sample k tries per prompt
    
        
    Data is the bottleneck, so make your own. For each prompt, sample several answers with temperature > 0 so they differ. One try might be wrong; among k tries, some are usually right — that's the raw material the filter will sift.
    
        
    def generate(model, prompts, k=8, temp=0.8):
    pool = []
    for p in prompts:                       # YOUR prompts / questions
        for _ in range(k):                  # k diverse attempts each
            pool.append((p, model.sample(p, temperature=temp)))
    return pool                             # many candidates, quality varies
# one shot is unreliable; k shots almost always contain a good one.
    
      
    2. 
      
  2. 
        
    2
    Verify & filter — the step that makes it all work
    
        
    The whole idea rests on verification being cheaper than generation. Prefer a ground-truth checker (unit tests, a math answer key, a compiler) over an LLM judge, and an LLM judge (Day 38) over nothing. Keep only what passes — the kept set is purer than the model's average output.
    
        
    def keep_good(pool, verify):
    kept = []
    for (prompt, answer) in pool:
        if verify(prompt, answer):          # tests pass? matches key? judge says OK?
            kept.append((prompt, answer))
    return kept
# verify() is the filter. ground-truth > LLM-judge > nothing.
# no verify() -> you retrain on your own average -> NO improvement.
    
      
    3. 
      
  3. 
        
    3
    Retrain on the kept set (STaR / rejection sampling)
    
        
    Fine-tune the model on its own verified-correct answers — including the reasoning that led there. This is STaR (Self-Taught Reasoner) / rejection-sampling fine-tuning: the model bootstraps from the good samples it can already occasionally produce.
    
        
    def retrain(model, kept):
    # supervised fine-tune on VERIFIED (prompt, answer) pairs
    return finetune(model, dataset=kept)    # keep the winning reasoning too
# the model learns from the best of its OWN attempts -> it improves
# at producing them on the first try next time.

  4. 
        
    4
    The loop — repeat, quality ticks up each round
    
        
    Chain the three steps and iterate. As the model gets better, more of its candidates pass the filter, so the kept set grows and improves — the virtuous cycle you drove with the filter slider in LOOK. It plateaus when the model saturates what the verifier can confirm.
    
        
    def self_improve(model, prompts, verify, rounds=5):
    for r in range(rounds):
        pool  = generate(model, prompts, k=8)
        kept  = keep_good(pool, verify)          # <-- the filter is everything
        model = retrain(model, kept)
        log(r, eval_quality(model))              # should rise each round
    return model
# filter ON  -> quality climbs.   filter OFF -> it flat-lines.

    Distillation variant — a teacher makes data for a student
    
        
    Same "generate data to train" family, different roles: a strong teacher produces high-quality answers (still verified/filtered), and a small cheap student trains on them. You bottle a big model's skill into a model you can actually afford to serve (Day 23).
    
        
    # self-improvement: teacher == student (learns from itself)
# distillation:     teacher  >  student (student mimics the teacher)
data    = keep_good(generate(teacher, prompts, k=4), verify)
student = finetune(small_student, dataset=data)      # cheap to serve
# synthetic data flows from a model that CAN do it -> one that will.

    Guard against collapse — keep real data, watch diversity
    
        
    The failure mode from LOOK ②: train gen-on-gen on unfiltered synthetic and the distribution narrows until it collapses (a.k.a. model autophagy / MAD). Defences: always mix in real/human data to anchor the tails, filter hard, cap the number of self-training generations, and track a diversity metric — stop if it drops.

train_set = mix(real_data, synthetic_kept, real_frac=0.3)   # anchor tails

When human data runs out, a model can make its own — because verifying is cheaper than generating. Generate many candidates, filter to the verified-good ones, and retrain: that's STaR / self-training, and distillation when a teacher feeds a student. The filter is the whole game — without it quality stalls, and training gen-on-gen on unfiltered output causes model collapse as diversity leaks away.
