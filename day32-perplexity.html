📉 Perplexity — the One Number That Says How "Surprised" a Language Model Is

A language model reads your sentence one token at a time and, before each one, says how likely it thought that token was. Perplexity rolls all those guesses into a single score: how surprised the model was, on average, per token.

For every token the model gives a probability P(token | the words so far). Turn each into surprisal — −log₂P bits — big when the token was unlikely, near zero when it was expected. Average the surprisal over the sentence and you have the cross-entropy; raise 2 to that power and you have PERPLEXITY = 2^(avg bits) = exp(avg negative log-likelihood). Lower is better — the model was less surprised. Type a fluent sentence (low PPL), a garbled one (high PPL), or a mostly-fluent one with a surprise word, and toggle a confident vs an uncertain model. Every probability, bit, and PPL below is real code running in your browser on a tiny scripted next-token model.

Perplexity is the standard intrinsic yardstick for a language model: average how surprised it is per token and report it as an effective branching factor. A PPL of 8 means the model was, on average, about as unsure as if it were picking uniformly among 8 words. It measures fit to the text distribution — not whether the text is true, useful, or safe.

🔨 Compute perplexity yourself

    
Perplexity is a five-line calculation on top of the probabilities a model already produces — collect the per-token probability of the correct token, take its negative log, average, exponentiate. Here's the whole thing, layer by layer, with the traps that quietly corrupt the number.

    
    
      
  1. 
        
    1
    Get the probability of each correct token
    
        
    Run the model over the text. At each position it outputs logits over the whole vocabulary; softmax turns them into a probability distribution. The one number you keep is the probability it assigned to the token that actually came next — that's P(tokenᵢ | token₁…tokenᵢ₋₁). Everything else follows from this list of probabilities.
    
        
    import torch, torch.nn.functional as F
logits = model(input_ids).logits[:, :-1, :]     # predict-next, drop last
targets = input_ids[:, 1:]                       # the actual next tokens
logp = F.log_softmax(logits, dim=-1)             # log-probs over the vocab
tok_logp = logp.gather(-1, targets.unsqueeze(-1)).squeeze(-1)  # log P(correct)
    
      
    2. 
      
  2. 
        
    2
    Negative log-likelihood, then average = cross-entropy
    
        
    Take the negative log of each probability (its surprisal) and average over all tokens. That average negative log-likelihood is exactly the cross-entropy loss the model was trained to minimize — perplexity and training loss are the same quantity, just reported differently. Work in log space: it turns a product of tiny probabilities into a sum and avoids numeric underflow.
    
        
    nll = -tok_logp                     # surprisal per token (nats, natural log)
cross_entropy = nll.mean()          # average NLL == the training loss
# this is identical to:
#   F.cross_entropy(logits.reshape(-1, V), targets.reshape(-1))
    
      
    3. 
      
  3. 
        
    3
    Exponentiate → perplexity
    
        
    Perplexity is just exp(cross_entropy) when you used natural log, or 2^(cross_entropy) when you measured in bits — the two are equal. Exponentiating undoes the log and puts the score back on a human scale: an effective branching factor. Keep your log base consistent end-to-end or the number is meaningless.
    
        
    ppl = torch.exp(cross_entropy)          # natural log throughout
# equivalently, in bits:
#   bits = nll / math.log(2)
#   ppl  = 2 ** bits.mean()
# a perplexity of 8  ->  "as unsure as picking among 8 equally-likely tokens"
    
      
    4. 
      
  4. 
        
    4
    Mask padding & sum correctly over long texts
    
        
    Two silent bugs. First, never average surprisal over <pad> tokens — they aren't predictions and will fake a low perplexity; mask them out. Second, for a document longer than the context window, slide the window and average the log-likelihoods over the whole corpus (sum the NLLs, divide by the true token count) before exponentiating once at the end.
    
        
    mask = (targets != PAD_ID)
total_nll = (nll * mask).sum()
n_tokens  = mask.sum()
ppl = torch.exp(total_nll / n_tokens)   # average over REAL tokens only
# long doc: accumulate total_nll & n_tokens across sliding windows, exp() once
    
      
    5. 
      
  5. 
        
    5
    Only compare on the same tokenizer — or use bits-per-byte
    
        
    Perplexity is per token, so it depends on how the text was split. A model with a bigger vocabulary uses fewer, "easier" tokens and looks better on raw PPL even if it's no smarter. Never compare perplexities across different tokenizers. To compare fairly, normalize to the text itself: bits-per-byte (total bits ÷ number of UTF-8 bytes) is tokenizer-independent.
    
        
    total_bits = total_nll / math.log(2)          # nats -> bits
bits_per_byte = total_bits / num_utf8_bytes    # tokenizer-independent
# raw PPL: only comparable when the tokenizer + eval text are identical
# leaderboards use bits-per-byte / bits-per-char for cross-model comparisons

  6. 
        
    6
    Read it right — and know what it can't tell you
    
        
    Lower perplexity means the model fits the text distribution better — great for tracking pretraining and picking checkpoints. But it measures fluency/fit, not truth, usefulness, or safety: a model can be confidently, fluently wrong and score a low perplexity. Use PPL as an intrinsic training/eval signal, and pair it with task benchmarks and human/LLM judgement for anything user-facing.
    
        
    # PPL answers: "how well does this fit the text?"  (intrinsic)
# PPL does NOT answer: is it true? helpful? safe? on-task?
# track PPL / bits-per-byte on a held-out set during pretraining;
# gate releases on downstream evals (MMLU, human prefs), not PPL alone.

🎉 Day 32 of AIFromZero done.
You saw the single most-used intrinsic metric for a language model, built from the ground up: a model gives P(next token | context), each token's surprisal is −log₂P bits, the average is the cross-entropy (the very loss it trains on), and perplexity = 2^(avg bits) = exp(avg NLL). Lower means less surprised means a better fit; the value reads as an effective branching factor — "about this many equally-likely choices." You learned why we use logs (products → sums, no underflow), why PPL is tokenizer-dependent (and how bits-per-byte fixes cross-model comparisons), and — crucially — what it does not measure: truth, usefulness, or safety. 👉 Tomorrow — Day 33: Beam Search vs Sampling — once you can score a sequence's probability, how do you actually pick one to generate?
