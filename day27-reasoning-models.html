🧠 Reasoning Models & Test-Time Compute — Think Longer, Answer Better

Same model, same question. Let it think longer and the wrong answer becomes right.

A normal model answers in one shot — it blurts the first plausible thing. A reasoning model (o1-style, DeepSeek-R1-style) is trained to spend inference compute first: it writes a long private chain-of-thought, checks its own work, catches its mistakes and backtracks, then commits. Drag the thinking budget below and watch a classic trap problem flip from wrong to right — and watch the accuracy-vs-compute curve climb, then flatten. Then flip on majority vote: sample several chains and let them vote. Every number here is computed live in your browser.

🔨 Spend test-time compute on purpose

    
A reasoning model isn't "smarter for free" — it's a dial. The skill is knowing when to turn it up, when to leave it down, and how to squeeze more accuracy out of the same weights with best-of-N and verification. Here's the practical loop, end to end.

    
    
      
  1. 
        
    1
    Pick the right model for the job
    
        
    Reasoning models cost more and answer slower. Route by task: a fast model for lookups, formatting, chit-chat; a reasoning model only for genuinely multi-step problems (math, planning, hard code, proofs). Don't pay the thinking tax on "what's the capital of France."
    
        
    def route(task):
    if task.kind in ("chat", "format", "lookup", "extract"):
        return "fast-model"          # cheap, low latency
    if task.kind in ("math", "plan", "hard-code", "multi-step"):
        return "reasoning-model"     # worth the compute
    return "fast-model"
    
      
    2. 
      
  2. 
        
    2
    Turn the dial with a reasoning-effort param
    
        
    Reasoning models expose a knob — reasoning_effort (low/medium/high) or an explicit thinking-token budget. Higher = longer internal chain-of-thought = more accuracy on hard items, at more cost. Start low, raise it only if the task needs it.
    
        
    resp = client.responses.create(
    model="reasoning-model",
    reasoning_effort="high",      # low | medium | high
    input=hard_problem,
)
# equivalently, some APIs take an explicit token budget:
#   thinking={"budget_tokens": 4000}
    
      
    3. 
      
  3. 
        
    3
    Don't over-prompt it to "think step by step"
    
        
    A reasoning model already reasons internally — hand-written "let's think step by step" scaffolding and few-shot chains often hurt. Give it a clean, complete problem statement and the constraints; let it do the thinking. Prompt for the output format, not the thought process.
    
        
    # fast model (pre-reasoning era): coax the CoT out by hand
prompt = "Let's think step by step...\n" + few_shot + question

# reasoning model: just state the task cleanly
prompt = question + "\nReturn only the final number."
    
      
    4. 
      
  4. 
        
    4
    Squeeze more out with best-of-N + majority vote
    
        
    When one chain isn't reliable enough, sample several and let them vote (self-consistency). This is the exact tally from the LOOK tab — a real argmax over sampled answers. Great accuracy lift for tasks with a checkable final answer; you pay N× the compute.
    
        
    from collections import Counter
def best_of_n(problem, n=5, temperature=0.7):
    answers = [sample(problem, temperature).final for _ in range(n)]
    counts  = Counter(answers)          # {"$0.05": 3, "$0.10": 1, ...}
    return counts.most_common(1)[0][0]  # majority-vote winner
    
      
    5. 
      
  5. 
        
    5
    Verify the answer — reasoning ≠ correct
    
        
    A long chain can still be confidently wrong. Where the answer is checkable, check it: run the code, evaluate the equation, unit-test the plan. Verifiable rewards are exactly what these models were trained on — reuse the same signal at inference to pick or reject an answer (best-of-N by verifier score, not just votes).
    
        
    cands = [sample(problem) for _ in range(n)]
scored = [(verify(c.answer, problem), c) for c in cands]   # 1.0 if it checks out
best = max(scored, key=lambda x: x[0])[1]
if verify(best.answer, problem) < 1.0:
    escalate()      # none verified -> raise effort or ask a human
    
      
    6. 
      
  6. 
        
    6
    Budget it: watch the accuracy-vs-cost curve
    
        
    Compute isn't free and the curve plateaus. Measure accuracy at a few effort levels on your own eval set, find the knee of the curve, and set the default there. Escalate to higher effort only for the items that fail — not for everything.
    
        
    for effort in ["low", "medium", "high"]:
    acc, cost, p95 = eval_suite(effort)
    print(f"{effort:6}  acc={acc:.2f}  $={cost:.3f}  p95={p95}ms")
# ship the effort at the KNEE: past it you pay a lot for ~nothing
default_effort = "medium"      # escalate only failing items to "high"

  7. 
        
    7
    Distill the reasoning into a smaller, faster model
    
        
    Once a big reasoning model solves your tasks, harvest its correct chains-of-thought and fine-tune a small model on them (Day 23 · distillation). The student learns to reason in fewer tokens — most of the accuracy, a fraction of the cost and latency.
    
        
    # collect verified reasoning traces from the big model
traces = [t for t in big_model_runs if verify(t.answer)]
# supervised fine-tune a small model to imitate the reasoning
small = finetune("small-base", data=[(t.problem, t.chain, t.answer) for t in traces])
# now: cheaper, faster, keeps most of the reasoning skill
