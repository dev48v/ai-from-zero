🛡️ Guardrails & Safety — The Filters That Sit Around a Tool-Using Model

The model from Day 28 can now act. Guardrails decide what's allowed to reach it — and what it's allowed to do back.

A guardrailed assistant is a model wrapped in two checkpoints. INPUT guards screen every request before the model sees it — moderation/blocklist, a prompt-injection detector, and PII redaction. The model then runs with least-privilege tools, and any risky action is routed to a human. OUTPUT guards then check the reply — scrub PII, validate format/topic, check groundedness — before it goes out. Pick a request, watch each guard return PASS / REDACTED / BLOCK / HOLD with its reason. Then flip guards off and watch the same payload cause real harm. Every check below is a real regex / rule running in your browser.

No single guard is enough — that's defense-in-depth. Moderation catches unsafe topics, the injection detector catches "ignore previous instructions", PII redaction protects data even on a benign request, the risky-tool gate stops irreversible actions, and the output guards catch anything the model leaks anyway. Turn guards off and every layer disappears at once.

🔨 Wrap a tool-using model in guardrails

    
Guardrails are a sandwich around the Day-28 tool loop: screen the input, run the model with least-privilege tools, gate anything risky on a human, then validate the output before it leaves. Here's the whole thing, layer by layer.

    
    
      
  1. 
        
    1
    Write the policy down first
    
        
    Guardrails are only as good as the policy behind them. Enumerate it explicitly: which content categories to refuse, which PII types to redact, what "on topic" means, an allow-list of tools, and which tools are risky (irreversible or costly). This config is the single source of truth every guard reads from.
    
        
    POLICY = {
  "block_categories": ["weapons", "violence", "self_harm", "illicit"],
  "pii_types":        ["EMAIL", "PHONE", "SSN", "CARD"],
  "allowed_topics":   ["shipping", "orders", "billing", "tracking"],
  "tools_allowed":    ["get_weather", "calculator", "track_order"],
  "tools_risky":      ["issue_refund", "delete_records", "send_email"],
}
    
      
    2. 
      
  2. 
        
    2
    Input guards — run them before the model
    
        
    Every request passes moderation, an injection detector, and PII redaction first. Moderation and injection can BLOCK outright; PII redaction rewrites the text so the model (and your logs) never see raw secrets. Fail closed — if a guard errors, refuse.
    
        
    def input_guards(text):
    if (cat := moderate(text)):
        return Block(f"unsafe category: {cat}")
    if (hit := detect_injection(text)):
        return Block(f"prompt injection: {hit}")
    clean, found = redact_pii(text)      # "john@x.com" -> "[REDACTED:EMAIL]"
    return Ok(clean, redacted=found)
    
      
    3. 
      
  3. 
        
    3
    Detectors: rules + a classifier model
    
        
    Cheap regex/blocklists catch the obvious ("ignore all previous instructions", an SSN pattern). But regex misses paraphrases, so back it with a small safety classifier (e.g. Llama Guard) that scores the text against your categories. Rules for precision, a model for recall — run both.
    
        
    INJECTION = [r"ignore (all|previous) instructions",
             r"(reveal|exfiltrate).*(system prompt|instructions)",
             r"you are now", r"developer mode|jailbreak"]

def detect_injection(text):
    if any(re.search(p, text, re.I) for p in INJECTION):
        return "rule match"
    return llama_guard(text).get("prompt_injection")   # model backstop
    
      
    4. 
      
  4. 
        
    4
    Gate risky tool calls on a human
    
        
    The model runs with the smallest tool set that does the job (least privilege). When it asks for a risky tool — a refund, a delete, an outbound email — don't execute it. Queue it for human approval and feed the model a "pending approval" result. "The model asked" is never authorization.
    
        
    for call in resp.tool_calls:
    if call.name not in POLICY["tools_allowed"] + POLICY["tools_risky"]:
        result = "BLOCKED: tool not on allow-list"
    elif call.name in POLICY["tools_risky"]:
        result = request_human_approval(call)   # HOLD, do not run
    else:
        result = TOOLKIT[call.name](**validate(call.arguments))
    messages.append(tool_msg(call.id, result))
    
      
    5. 
      
  5. 
        
    5
    Output guards — validate before it leaves
    
        
    The reply is untrusted too. Scrub any PII the model echoed, validate the shape (JSON matches the schema, on an allowed topic), and check groundedness — every factual claim should trace to a tool result, not a hallucination. On failure, rewrite or refuse rather than ship it.
    
        
    def output_guards(reply, tool_results):
    reply, _ = redact_pii(reply)                 # scrub leaked PII
    if not validates(reply, RESPONSE_SCHEMA):
        return Block("malformed / off-schema output")
    if moderate(reply):
        return Block("unsafe content in reply")
    if not grounded(reply, tool_results):        # claims must trace back
        return Rewrite("drop unsupported claims")
    return Ok(reply)
    
      
    6. 
      
  6. 
        
    6
    Decide, log, and lean on a library
    
        
    Collapse the layers into one decision — allow / block / rewrite / hold — log every verdict for audit, and fail closed on any error. You rarely hand-roll all of this: NeMo Guardrails, Guardrails AI, and Llama Guard give you the detectors, validators and a policy DSL out of the box.
    
        
    ig = input_guards(user_text)
if ig.blocked: return refuse(ig.reason, log=True)

resp   = run_model(ig.clean, tools=least_privilege_tools)
og     = output_guards(resp.text, resp.tool_results)

return {"allow": og.ship, "block": refuse, "rewrite": og.fixed,
        "hold": pending_approval}[og.decision]     # fail closed on error
