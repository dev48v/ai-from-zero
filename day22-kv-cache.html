🗄️ The KV-Cache — Why LLMs Don't Re-Read the Prompt

Generate a sentence twice — once re-reading everything, once with a cache.

To pick each new word, a Transformer's attention looks back at every previous token. The naïve way recomputes the Key and Value vectors for all prior tokens at every single step — that's quadratic, and most of it is repeated work. The KV-cache stores each token's K/V the first time and reuses them, so each new step computes K/V for just one new token. Press ▶ and watch the operation counters diverge.

Without a cache, generating N tokens costs work proportional to 1+2+3+…+N = N(N+1)/2 (quadratic). With the cache it's proportional to N (linear) — each step reuses all prior K/V and computes only the newest. The blue slots are freshly computed; the light slots are cache hits (free). The cache is also why long contexts are expensive: it grows with every token and lives in GPU memory.

🔨 The mental model

    
You rarely implement the KV-cache yourself — the inference engine does — but here's exactly what it's doing inside the decode loop.

    
    
      
  1. 
        
    1
    Attention needs every past token's K & V
    
        
    For the new token's Query, attention scores it against the Key of every previous token, then mixes their Values. So to generate token t you need K and V for tokens 1…t.
    
        
    # to produce the next token, attention does:
scores = Q_new · K[1..t]ᵀ         # compare against all past keys
weights = softmax(scores)
out = weights · V[1..t]           # blend all past values
    
      
    2. 
      
  2. 
        
    2
    The naïve loop recomputes everything
    
        
    Without a cache, each step re-runs the whole sequence through the layers to rebuild K/V for tokens 1…t. Token 1's K/V is recomputed N times. Total work ≈ N²/2.
    
        
    # NAIVE: O(N²) — recompute K/V for the full prefix each step
for t in range(N):
    K, V = model.encode(tokens[:t+1])   # ← redoes tokens 1..t every time
    next = attend(Q_t, K, V)
    
      
    3. 
      
  3. 
        
    3
    Cache K/V, compute only the new token
    
        
    Past tokens never change under causal masking, so their K/V never change either. Store them once; each step appends just the new token's K/V and attends over the whole cache. Work per step is constant → O(N) total.
    
        
    # CACHED: O(N) — append one token's K/V per step
K_cache, V_cache = [], []
for t in range(N):
    k, v = model.kv(token_t)        # ← ONE token only
    K_cache.append(k); V_cache.append(v)
    next = attend(Q_t, K_cache, V_cache)   # reuse all prior K/V
    
      
    4. 
      
  4. 
        
    4
    The memory cost of long context
    
        
    The cache stores K and V for every token, every layer, every attention head. Its size grows linearly with context length — which is exactly why a 128k-token context is expensive: the KV-cache can consume many gigabytes of GPU memory. Tricks like paged attention (vLLM), grouped-query attention, and quantised caches exist to tame it.
    
        
    # KV-cache memory ≈
#   2 (K and V) × layers × heads × head_dim
#   × context_length × bytes_per_value
# grows with context_length → why 128k context is pricey
