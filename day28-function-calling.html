🛠️ Function Calling & Tool Use — Let the Model Act, Not Just Talk

The model can't multiply or check the weather. So it asks for a tool — and your code runs it.

Give a model a menu of tools (each a JSON schema) and it stops guessing: instead of prose, it returns a structured tool call {name, arguments}. Your code runs the real function, feeds the result back, and the model continues — call → run → feed back → call — until it writes the final answer. Step through the loop below. The calculator is a real, live arithmetic evaluator (no eval()); the weather tool is mocked. Flip between sequential and parallel tool calls and watch the messages[] array grow.

🔨 Wire up a tool-calling loop

    
Function calling is a small, mechanical loop you own end to end: describe your tools, let the model request one, run it, feed the result back, repeat until it stops asking. Here's the whole thing, plus the guards that keep it safe.

    
    
      
  1. 
        
    1
    Define your tools as JSON schemas
    
        
    One schema per tool: a clear name, a description the model can actually act on, and typed parameters with the required ones listed. Keep the set small and unambiguous — overlapping or vague tools make the model pick wrong. Map each name to the real function in a lookup table.
    
        
    TOOLS = [
  {"name": "calculator",
   "description": "Evaluate an arithmetic expression and return the number.",
   "parameters": {"type": "object",
     "properties": {"expr": {"type": "string", "description": "e.g. 23*47"}},
     "required": ["expr"]}},
  {"name": "get_weather",
   "description": "Get the current weather for a city.",
   "parameters": {"type": "object",
     "properties": {"city": {"type": "string"}},
     "required": ["city"]}},
]
TOOLKIT = {"calculator": calculator, "get_weather": get_weather}
    
      
    2. 
      
  2. 
        
    2
    Send the tools with every model call
    
        
    Pass the schemas alongside the messages. The model can now either answer in plain text or return a tool call — you don't tell it which; it decides from the descriptions and the user's need.
    
        
    messages = [{"role": "user", "content": user_question}]
resp = client.chat(model="...", messages=messages, tools=TOOLS)
    
      
    3. 
      
  3. 
        
    3
    Detect a tool call in the response
    
        
    Check whether tool_calls came back (the finish reason will say so). No calls → the model gave a final answer and you're done. Calls present → it wants you to run something before it can continue.
    
        
    if not resp.tool_calls:
    print(resp.content)      # final answer -> stop here
else:
    ...                      # there is work to do -> run the tools
    
      
    4. 
      
  4. 
        
    4
    Validate, run, and feed each result back
    
        
    Append the assistant's tool_calls turn first so history stays paired. Then for each call: validate the arguments against the schema, execute the matching function, and append the return value as a tool-role message keyed by tool_call_id. On failure, feed the error back instead of crashing.
    
        
    messages.append(resp.assistant_message)              # the tool_calls turn
for call in resp.tool_calls:
    try:
        args   = validate(call.arguments, schema_for(call.name))
        result = TOOLKIT[call.name](**args)          # YOU run it
    except Exception as e:
        result = f"ERROR: {e}"                        # errors are feedback
    messages.append({"role": "tool", "tool_call_id": call.id,
                     "content": str(result)})
    
      
    5. 
      
  5. 
        
    5
    Loop until it stops — with a hard guard
    
        
    Call the model again with the grown history and repeat: call → run → feed back → call. The natural stop is when the model returns a normal message with no tool calls. Always cap the iterations too, so a misbehaving model can't loop forever.
    
        
    for _ in range(MAX_STEPS):                 # hard stop = guardrail
    resp = client.chat(model="...", messages=messages, tools=TOOLS)
    if not resp.tool_calls:
        return resp.content                # natural stop: final answer
    run_and_append(resp, messages)         # step 4
raise RuntimeError("tool loop did not converge")

    Guard risky tools & validate every argument
