AIFromZero · Day 40

📈 Scaling Laws — a model's loss falls in a smooth, predictable power law as you pour in more compute, data and parameters. The surprise isn't that bigger is better; it's that the curve is a straight line on a log-log plot — so you can forecast a giant run before you pay for it, and split your budget the Chinchilla-optimal way: ~20 tokens per parameter.

Bigger models get better — but how much better, and where should the money go?

Two of the most useful facts in modern AI. First: test loss drops as a clean power law in compute — a straight line on a log-log plot — down to an irreducible floor you can never cross. That means you can extrapolate: fit a few small runs, draw the line, and predict how a 100×-bigger model will do before training it. Second: for a fixed compute budget there's a best way to split it between model size (N parameters) and data (D tokens). Kaplan (2020) said pour it into parameters; Chinchilla (2022) showed you should grow both together — about 20 tokens per parameter — which is why GPT-3 was badly undertrained. Both demos below compute the real curves in your browser, no model calls.

① The power law of loss — L = L₀ + (C / C_c)−α · drag compute along the curve and watch loss fall in a straight log-log line toward the floor

Loss has two parts: a reducible piece that shrinks as a power of compute (the dashed straight line), and an irreducible floor L₀ — the entropy of language itself, which no amount of scale removes. Steeper α = you learn faster per FLOP. As compute grows the curve rides the power law, then bends flat onto the floor: that's diminishing returns and the limit of extrapolation.

② The Chinchilla allocator — fixed compute C ≈ 6·N·D · slide the budget and the split, find the loss-minimising ~20 tokens/param

A training run costs roughly C = 6 · N · D FLOPs (params × tokens). For a fixed C you trade model size against data: fewer params ⇒ more tokens, and vice-versa. Loss is U-shaped in that split — a clear minimum near 20 tokens per parameter. Go left (too many params, too little data) and you're undertrained — a big model starved of data (that was Kaplan's regime, and GPT-3). Go right and you've spent on data a bigger model would have used better. (y-axis zoomed — the valley is gentle near the bottom, but veering far from it wastes real compute.)

🦖 “GPT-3 was undertrained.” 175B params but only ~300B tokens ≈ 1.7 tokens/param — deep in the undertrained zone. Chinchilla later trained 70B on 1.4T tokens (20 : 1) on the same compute and beat GPT-3 — a model 2.5× smaller and far cheaper to serve. Click 🦖 above to jump the marker there.

Two laws, one lesson. Card ① says loss is a predictable power law — you can forecast a frontier run and know where the irreducible floor makes more compute pointless. Card ② says a fixed budget has a compute-optimal split (~20 tokens/param): under-feed a big model and you waste it (Kaplan / GPT-3), over-feed a small one and you leave capability on the table. Scaling isn't just “make it bigger” — it's “grow N and D together, and know when the curve flattens.”

🔨 Build a scaling-law forecaster & a Chinchilla budget planner

    
Turn a handful of small training runs into a forecast for a giant one, then split a compute budget the compute-optimal way. Fit a power law in log-log space, read off the exponent and the irreducible floor, and use C = 6ND to solve for the best N and D — the exact logic behind LOOK ① and ②.

    
    
      
  1. 
        
    1
    Fit the power law — a straight line in log-log space
    
        
    Run a few small models at different compute budgets and record final loss. Subtract the irreducible floor, take logs of both axes, and fit a line: its slope is −α. The whole point of scaling laws is that these points really do fall on a line.
    
        
    import numpy as np
# small, cheap runs: (compute C in FLOPs, final test loss)
C    = np.array([1e17, 3e17, 1e18, 3e18, 1e19])
loss = np.array([3.10, 2.74, 2.45, 2.22, 2.04])

L0   = 1.69                       # irreducible floor (fit or assume)
y    = np.log(loss - L0)          # reducible loss, logged
x    = np.log(C)
alpha, logCc = np.polyfit(x, y, 1)   # straight-line fit
alpha = -alpha                    # slope is -alpha
# loss(C) = L0 + (C / Cc)^(-alpha)   <-- your fitted law
    
      
    2. 
      
  2. 
        
    2
    Extrapolate — forecast a run you haven't paid for
    
        
    With α, C_c and the floor L₀ you can predict the loss of a model 100× or 1000× bigger — the trick GPT-4's team used to de-risk a nine-figure run. But respect the floor: as C grows the reducible term vanishes and predictions asymptote to L₀. Extrapolate too far and the law itself may break.
    
        
    def predict_loss(C, L0, Cc, alpha):
    return L0 + (C / Cc)**(-alpha)   # the fitted power law

predict_loss(1e24, L0, Cc, alpha)    # forecast a frontier run
# reducible part -> 0 as C grows, so loss -> L0 (the floor).
# never predict BELOW L0, and treat far extrapolations with care.

  3. 
        
    3
    The Chinchilla loss surface — L(N, D)
    
        
    Loss depends on both size and data. Chinchilla fit a surface: an irreducible term plus one power law in N (parameters) and one in D (tokens). Neither term alone wins — starve either and the loss climbs.
    
        
    # Hoffmann et al. 2022 (Chinchilla) parametric fit:
#   L(N, D) = E + A / N^alpha + B / D^beta
E, A, B      = 1.69, 406.4, 410.7
alpha, beta  = 0.34, 0.28

def loss(N, D):
    return E + A / N**alpha + B / D**beta
# E = irreducible. A/N^a shrinks with size; B/D^b shrinks with data.
# BOTH terms must be small -> you must grow N and D together.

  4. 
        
    4
    Solve the budget — the ~20 tokens/param rule
    
        
    Fix compute at C = 6ND and minimise the loss surface over the N/D split. The optimum grows N and D at nearly the same rate, so their ratio D/N stays roughly constant — about 20 tokens per parameter. That single number is the headline Chinchilla result (the demo's engine).
    
        
    import numpy as np
def optimal_split(C):
    best = None
    for r in np.logspace(-1, 3, 400):     # try many tokens/param ratios
        N = np.sqrt(C / (6*r))            # since C = 6*N*D and r = D/N
        D = np.sqrt(C * r / 6)
        L = loss(N, D)
        if best is None or L < best[0]:
            best = (L, N, D, r)
    return best                           # r* lands near ~20 : 1

L, N, D, r = optimal_split(6e23)          # -> ~70B params, ~1.4T tokens

    Diagnose a run — Kaplan vs Chinchilla
    
        
    Given any real model, check its ratio. Far below 20 ⇒ undertrained (Kaplan's “make it bigger” era, and GPT-3). Far above ⇒ over-fed a model too small to use the data. Modern models (Llama-style) deliberately run past 20 because they optimise for cheap inference, not just training-optimal loss.
    
        
    def diagnose(N, D):
    r = D / N
    if r < 12:  return f"{r:.1f} tok/param — UNDERTRAINED (Kaplan/GPT-3)"
    if r > 32:  return f"{r:.1f} tok/param — data-heavy (inference-optimal)"
    return       f"{r:.1f} tok/param — near Chinchilla-optimal"

diagnose(175e9, 300e9)   # GPT-3  -> 1.7  tok/param, undertrained
diagnose( 70e9, 1.4e12)  # Chinchilla -> 20 tok/param, optimal
# Kaplan 2020: scale N.  Chinchilla 2022: scale N and D together.

    Know the limits — floors, data walls, breakdowns
    
        
    Scaling laws are a forecast, not a guarantee. Loss can't go below the irreducible floor; the Chinchilla-optimal recipe eventually demands more tokens than exist (the data wall — Day 39's synthetic data is one answer); laws are fit for loss, not downstream skill; and the fit can bend or break far outside the range you measured.

assert predicted_loss > L0, "cannot beat the irreducible floor"

assert tokens_needed < tokens_available, "you hit the DATA WALL"

🎉 Day 40 of AIFromZero done.
Test loss falls as a smooth power law in compute, data and parameters — a straight line on a log-log plot down to an irreducible floor — which lets you forecast a frontier run before paying for it. And for a fixed budget C ≈ 6ND there's a compute-optimal split: grow N and D together at about 20 tokens per parameter. That's why Kaplan's “just make it bigger” gave way to Chinchilla, and why GPT-3 (1.7 tok/param) was undertrained — a smaller model on more data matched it. Mind the limits: the floor, the data wall, and loss ≠ capability. 👉 Tomorrow — Day 41: Emergent Abilities — why some skills appear suddenly as models scale (and whether that jump is real or a mirage of how we measure it).
