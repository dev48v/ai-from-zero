💾 Prompt Caching — Compute a Long Prefix Once, Reuse It Cheap on Every Call

Every API call re-reads your whole prompt from scratch. When the first 90% of it never changes — a long system prompt, a document, few-shot examples — prompt caching computes that part once and reuses it for pennies on every later call.

A prompt is a big stable prefix (system + doc + few-shot) followed by a small changing suffix (the user's question). The first call is a cache MISS: the model runs the expensive prefill over the whole prefix and stores the result. Every later call with a byte-identical prefix is a cache HIT: the prefix is reused, only the new question is processed — cached input is billed at ~10% and the prefill latency disappears. Edit even one character of the prefix and the cache breaks — you're back to a miss. Send a few requests, edit the prefix, watch the cost/latency meters and the TTL countdown. Every token count, cost and hit/miss below is real code running in your browser.

🔨 Make a long, repeated prompt cheap

    
Prompt caching is one design decision — split the prompt into a stable prefix and a volatile suffix, put the stable part first, and mark it — plus a bit of cost math. Here's the whole thing, layer by layer.

    
    
      
  1. 
        
    1
    Split the prompt: stable prefix vs volatile suffix
    
        
    Look at what changes between calls. The system prompt, tool definitions, a long document, and few-shot examples usually don't change — that's your prefix. The user's question changes every call — that's your suffix. Caching only pays off when the prefix is large and reused, so this split is the whole game.
    
        
    STABLE PREFIX  (same every call → cache it)
  system prompt + tool defs + long doc + few-shot

VOLATILE SUFFIX  (new every call → never cached)
  the user's actual question
    
      
    2. 
      
  2. 
        
    2
    Order for cache-friendliness — stable first
    
        
    The cache is a prefix match: it works from the start of the prompt up to a breakpoint. A byte change anywhere invalidates everything after it, so anything volatile must come last. Never interpolate datetime.now(), a request ID, or the user's name into the system prompt — that changes the prefix every call and nothing ever caches.
    
        
    # render order is: tools → system → messages
# GOOD: volatile bits live at the very end
system = FROZEN_SYSTEM_PROMPT          # never changes
messages = [{"role":"user","content": todays_question}]

# BAD: a timestamp at the front breaks caching every call
system = f"Today is {datetime.now()}. " + FROZEN_SYSTEM_PROMPT
    
      
    3. 
      
  3. 
        
    3
    Mark the breakpoint
    
        
    Tell the API where the stable prefix ends. On Anthropic you add an explicit cache_control breakpoint on the last stable block; OpenAI caches long prefixes automatically. Either way the prefix must clear a minimum length (~1,024–4,096 tokens depending on model) or it silently won't cache.
    
        
    r = client.messages.create(
    model="claude-opus-4-8", max_tokens=512,
    system=[{
        "type": "text",
        "text": LONG_PREFIX,                 # frozen system + doc + few-shot
        "cache_control": {"type": "ephemeral"}   # ← breakpoint (5-min TTL)
    }],
    messages=[{"role": "user", "content": question}],
)
    
      
    4. 
      
  4. 
        
    4
    Detect hits and misses
    
        
    Never assume it worked — read the usage fields. cache_creation_input_tokens is what you wrote (first call, the miss); cache_read_input_tokens is what you reused (the hits). If reads stay at zero across identical-prefix calls, a silent invalidator is at work — a timestamp, unsorted JSON keys, or a changing tool set in the prefix.
    
        
    u = r.usage
print(u.cache_creation_input_tokens)  # written to cache (~1.25x cost, once)
print(u.cache_read_input_tokens)      # served from cache (~0.1x cost)
print(u.input_tokens)                 # the uncached remainder (full price)
# reads stuck at 0? diff the rendered prompt bytes between two calls.
    
      
    5. 
      
  5. 
        
    5
    Do the cost math — when it actually pays
    
        
    A cache write costs a premium (~1.25× for the 5-min TTL, ~2× for the 1-hour TTL); a cache read costs ~0.1×. So the first call is slightly more expensive and every reuse is ~90% cheaper on the cached tokens. With the 5-min TTL you break even on the second call; the 1-hour TTL needs a third. Cache only what's reused enough to clear that bar.
    
        
    read_cost  = 0.10 * base_input_price   # ~90% cheaper — the win
write_cost = 1.25 * base_input_price   # paid once, on the first (miss) call
# 5-min TTL: 1.25x + 0.10x  < 2x  → break even on call #2
# 1-hour TTL: 2.00x write   → needs ~3 calls to pay off

  6. 
        
    6
    Keep the prefix byte-identical · TTL, pre-warm, providers
    
        
    Freeze the prefix: sort JSON keys deterministically, don't swap tools or models mid-conversation, and inject dynamic context after the breakpoint (or as a mid-conversation system message). Cache entries expire on a TTL (5 min or 1 hour) and evict under pressure — pre-warm with a cheap call if first-request latency matters. Anthropic uses explicit cache_control breakpoints; OpenAI/others cache automatically — the design is the same either way.
    
        
    import json
prefix = SYSTEM + json.dumps(config, sort_keys=True) + DOC  # deterministic
# don't: change the tool list, switch model, or edit the prefix mid-session
# do: put per-request context after the breakpoint; pre-warm on startup
# ttl: {"type":"ephemeral"} = 5 min · {"type":"ephemeral","ttl":"1h"} = 1 hour

🎉 Day 31 of AIFromZero done.
You saw why prompt caching is the highest-leverage cost fix for any app with a long, repeated prefix: the model runs the expensive prefill over the stable prefix once, stores its state, and on every later call with a byte-identical prefix serves it back at ~10% of the price with the prefill latency gone. You learned the split (stable prefix vs volatile suffix), why order matters (prefix match — stable first), the write/read cost math, TTL and eviction, and the provider split (Anthropic explicit breakpoints vs OpenAI automatic).
