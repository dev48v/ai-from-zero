🔧 LoRA Fine-Tuning — Freeze the Giant, Train Two Tiny Matrices

Adapt a frozen weight matrix by learning a skinny detour — W + BA.

Full fine-tuning updates every weight in a giant matrix W and stores a whole new copy. LoRA freezes W (🔒 never touched) and learns a small correction ΔW = B·A, where A is r×d and B is d×r. Drop the rank r and the number of trainable weights collapses from d² to 2·r·d — often under 1% of the model. Everything below is real: a real frozen matrix, real B·A math, and real gradient descent on A and B to make W + BA match a target.

1 · The parameter budget

Base matrix W is 8×8 = 64 weights, all 🔒 frozen. LoRA trains only A (r×8) and B (8×r) = 2·r·d weights. Slide the rank.

Trainable = 2 · r · d (A has r·d, B has d·r). Scaling α/r multiplies the detour so you can raise the update strength without touching the base. As r falls the bar shrinks — that's the whole point: adapt the model by moving a fraction of a percent of its weights.

2 · W + (α/r)·B·A, live

The frozen base W, the learned detour (α/r)·BA, and their sum — the effective weights the model actually uses at inference. Train below and watch the middle matrix fill in.

Each square is one weight; blue = positive, red = negative, brighter = larger. Only the middle matrix is ever trained — and it's forced to be low-rank because it's built from just A and B.

3 · Train ONLY A and B (W stays frozen)

We want the adapted matrix W + (α/r)·BA to equal a fixed target. Only A and B get gradients; W never moves. Pick the target's true rank — LoRA fits a low-rank change easily, but struggles when the needed change is higher-rank than r.

The curve is the real mean-squared error between W + (α/r)·BA and the target, one point per gradient step. When the target's true rank is ≤ your adapter rank r, the loss drives to nearly zero — a skinny detour is enough. Set the target to full-rank (8) with a small r and the loss bottoms out well above zero: the adapter physically can't express a change richer than rank r. That single fact is why LoRA works so well for most fine-tunes and occasionally needs a bigger r.

🔨 Fine-tune a giant model with LoRA yourself

    
You don't touch the base weights at all. You attach tiny adapter matrices to a few layers, train only those, and either keep them as a swappable file or merge them back in. Here's the whole recipe, ending in the one config that does it in Hugging Face peft.

    
    
      
  1. 
        
    1
    Load the base model & freeze it
    
        
    Load your pretrained model normally, then freeze every parameter. Nothing in the giant base will receive a gradient — it's read-only. This is what lets a 7B+ model be "fine-tuned" on a single consumer GPU.
    
        
    from transformers import AutoModelForCausalLM
model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3-8B")
for p in model.parameters():
    p.requires_grad = False          # base is frozen 🔒
    
      
    2. 
      
  2. 
        
    2
    Understand the shapes: A, B, and the sum
    
        
    For a weight matrix of shape d×k, LoRA adds B·A where A is r×k and B is d×r. A starts random, B starts at zero — so at step 0 the adapter is a no-op and the model behaves exactly like the base. Trainable count is r·k + d·r, i.e. 2·r·d when square.
    
        
    d, k, r = 4096, 4096, 8
A = randn(r, k) * 0.01     # r×k, small random
B = zeros(d, r)            # d×r, starts at ZERO → ΔW = 0 at start
# effective weight = W + (alpha / r) * (B @ A)
# trainable = r*k + d*r  =  2*r*d  (≈ 65k vs 16.7M full)
    
      
    3. 
      
  3. 
        
    3
    Pick which layers get an adapter
    
        
    You rarely adapt every layer. The classic choice is the attention projection matrices (q_proj, v_proj); adding k_proj, o_proj, and the MLP layers gives more capacity at more cost. Fewer target modules = fewer trainable weights = faster and lighter.
    
        
    target_modules = ["q_proj", "v_proj"]   # start small
# more capacity, more params:
# ["q_proj","k_proj","v_proj","o_proj","gate_proj","up_proj","down_proj"]
    
      
    4. 
      
  4. 
        
    4
    Train only the adapters
    
        
    Run an ordinary training loop. Because only A and B require gradients, the optimizer state is tiny and memory drops dramatically. The forward pass computes Wx + (α/r)·B(Ax); the backward pass only flows into A and B.
    
        
    opt = torch.optim.AdamW(
    [p for p in model.parameters() if p.requires_grad], lr=2e-4)
for x, y in loader:
    loss = model(x, labels=y).loss
    loss.backward(); opt.step(); opt.zero_grad()   # only A,B move
    
      
    5. 
      
  5. 
        
    5
    Save a tiny adapter — or merge it back
    
        
    The trained adapter is often a few megabytes, not gigabytes — you can keep many per base model and hot-swap them per task. Or, for zero inference latency, fold (α/r)·BA into W once and ship a normal-looking model with no extra ops at runtime.
    
        
    model.save_pretrained("./adapter")       # few MB, swappable
# --- or bake it in for zero-latency serving ---
merged = model.merge_and_unload()        # W <- W + (alpha/r)*B@A
merged.save_pretrained("./merged-model")  # plain model, no LoRA ops
    
      
    6. 
      
  6. 
        
    6
    The whole thing in peft: LoraConfig
    
        
    In practice you never hand-write A and B. Hugging Face peft wraps your model, injects the adapters into the target modules, and freezes everything else. This one config is the entire mechanism from tabs 1–3, productionised.
    
        
    from peft import LoraConfig, get_peft_model

config = LoraConfig(
    r=8,                       # adapter rank (the slider)
    lora_alpha=16,             # scaling: effective = W + (alpha/r)*B@A
    target_modules=["q_proj", "v_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)
model = get_peft_model(model, config)
model.print_trainable_parameters()
# -> trainable params: 4.2M || all params: 8.03B || trainable%: 0.05
