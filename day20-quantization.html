🗜️ Quantization — Run Big Models on Small Hardware

Squeeze a model's weights into fewer bits — watch the size shrink and the numbers snap to a grid

Every weight in a model is just a number. At full precision (FP32) each one takes 32 bits = 4 bytes. Pick a smaller bit-width below and the demo really quantizes: it maps the weights' min…max range onto a small set of 2bits levels, rounds every weight to its nearest level, and shows the rounding error. Watch the continuous values snap onto the grid, the memory readout collapse, and the quality dip a little at INT4. Same model — ~4–8× smaller and faster.

Full precision: nothing is lost, but you need the most memory and the biggest GPU.

Quantization stores each weight in fewer bits (32 → 8 → 4) — the same model gets roughly 4–8× smaller and faster with only a small quality drop, which is how big LLMs run on a laptop, phone, or a single GPU. The numbers above are a real round-trip: weights are mapped to a fixed number of integer levels, rounded, and dequantized — the error you see is exactly the precision you traded away for size and speed. Real LLM quantizers do this per-channel or per-block (not over the whole tensor) and protect a few outliers, so quality holds up better than this whole-tensor toy.

🔨 Quantize weights yourself, then load a model in 4-bit

    
Quantization is a tiny bit of arithmetic: find the range of the weights, split it into a fixed number of integer levels, snap each weight to the nearest level, and remember a scale (and zero-point) so you can turn the integers back into floats. Below you build that round-trip from scratch, measure the error and the size win, then see the one-liners real tools use — bitsandbytes load_in_4bit and a GGUF Q4_K_M file for llama.cpp. Copy each step as a starting point.

    You now know what quantization is — storing each weight in fewer bits by mapping a float range onto a small set of integer levels with a scale — why it matters (7B params × 4 bytes = 28 GB in FP32, but only ~7 GB in INT8 and ~3.5 GB in INT4), how it works (scale = range / (2bits−1), q = round((w−min)/scale), dequant w′ = q·scale + min), where the small quality drop comes from (rounding error, worst at INT4), how real models keep quality up (per-channel / per-block scales, outlier protection, mixed precision), and the tools that do it for you — GGUF / llama.cpp, GPTQ, AWQ, bitsandbytes.
