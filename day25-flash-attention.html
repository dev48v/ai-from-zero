⚡ Flash Attention — Exact Attention Without the N×N Blow-Up

Compute the exact same attention — but never store the giant score matrix.

Standard attention builds the full N×N matrix of scores QKᵀ in memory, softmaxes it, then multiplies by V. That matrix grows quadratically — double the context, quadruple the memory and the slow reads/writes to GPU main memory (HBM). Flash Attention streams over tiles of K and V, keeping a running max and running sum (online softmax) and an accumulating output, so the full matrix is never materialized. Same math, same result to the last bit — dramatically less memory and traffic. Everything below is real: real online-softmax arithmetic and a real numeric check that both methods give the identical answer.

1 · Standard attention — materialize N×N

For sequence length N, every query scores against every key: an N×N grid is written to memory, softmaxed, then read back. Slide N and watch the grid — and the memory — balloon.

Memory grows as N². This is the whole problem: at real context lengths the score matrix dwarfs the model itself.

2 · Flash attention — stream one tile

Process a query block, then stream over K/V blocks. Only one small tile lives in fast on-chip memory at a time; a running max m and sum l carry the softmax forward. Slide the block size and watch the tile sweep.

Block B 8
Scores stored
64
= B² (constant in N)
HBM traffic
288
≈ read Q,K,V + write O

The score tile is B×B — flat as N grows. Peak working memory is 9× smaller than the full matrix here, and the gap widens with every extra token.

At a realistic N = 8,192 tokens, the standard score matrix is 67,108,864 numbers ≈ 128 MB per head, per layer — written to and read from slow HBM every forward pass. Flash Attention keeps a fixed B×B tile on-chip and never writes that matrix at all. That is the entire speed and memory win.

4 · The online softmax, block by block

One query, 6 keys, streamed in blocks of 2. Each block we update the running max m, rescale the old totals by exp(mold−mnew) so nothing overflows, add the new block, and carry forward a running sum l and output O. Step through and watch it land exactly on the true softmax.

🔨 Use Flash Attention yourself

    
You almost never hand-write the kernel — but understanding the tiling loop makes the one-line API obvious. Here's the conceptual algorithm, then the two ways you actually turn it on in practice: PyTorch's built-in scaled_dot_product_attention, and the dedicated flash-attn package.

    
    
      
  1. 
        
    1
    The baseline you're replacing
    
        
    Standard attention allocates the full N×N score tensor, softmaxes it, then multiplies by V. It's simple and correct — and it's the exact thing that blows up your memory at long context. This is the code Flash Attention gives the same answer as.
    
        
    import torch, torch.nn.functional as F
def naive_attention(Q, K, V):        # Q,K,V: [batch, heads, N, d]
    d = Q.size(-1)
    scores = (Q @ K.transpose(-2, -1)) / d**0.5   # [.., N, N]  <-- materialized!
    weights = F.softmax(scores, dim=-1)           # [.., N, N]  <-- again in HBM
    return weights @ V                            # [.., N, d]

    2
    The tiling loop, in pseudocode
    
        
    This is what the kernel does under the hood: for every query block, stream over key/value blocks keeping a running max m, running sum l, and accumulator O. The N×N matrix never exists — only one B×B tile at a time.
    
        
    # for each query block Qi (size Bq):
for i in range(0, N, Bq):
    m = -inf; l = 0; O = zeros(Bq, d)          # running stats (on-chip)
    for j in range(0, N, Bk):                  # stream K/V blocks
        S  = Qi @ Kj.T / sqrt(d)               # B×B tile only
        m_new = max(m, rowmax(S))
        P  = exp(S - m_new)                     # numerically stable
        corr = exp(m - m_new)                   # rescale the past
        l  = corr*l + rowsum(P)
        O  = corr*O + P @ Vj
        m  = m_new
    Oi = O / l                                  # normalize once, at the end

    The easy way: PyTorch SDPA
    
        
    Since PyTorch 2.0, F.scaled_dot_product_attention automatically dispatches to a fused Flash-Attention kernel when your shapes/dtype/hardware allow. One call replaces steps 1–2 — no matrix materialized, no math to maintain. This is what you should reach for first.
    
        
    import torch.nn.functional as F
# drop-in, fused, memory-efficient — uses Flash Attention when eligible
out = F.scaled_dot_product_attention(Q, K, V, is_causal=True)

# force/inspect the backend:
from torch.nn.attention import sdpa_kernel, SDPBackend
with sdpa_kernel(SDPBackend.FLASH_ATTENTION):
    out = F.scaled_dot_product_attention(Q, K, V, is_causal=True)

    The dedicated package: flash-attn
    
        
    For maximum control (variable-length sequences, packed batches, the newest FA-2/FA-3 kernels) install the official package from Tri Dao's lab. Note the memory layout: it expects [batch, seqlen, heads, dim], not the SDPA [batch, heads, seqlen, dim].
    
        
    pip install flash-attn --no-build-isolation

    from flash_attn import flash_attn_func
# q,k,v: [batch, seqlen, n_heads, head_dim]
out = flash_attn_func(q, k, v, causal=True)   # exact, IO-aware, fast

    Turn it on inside Hugging Face
    
        
    You rarely call the kernel directly — you flip a flag when loading a model. Transformers wires Flash Attention into every attention layer for you, which is where most people actually get the win at training and inference time.
    
        
    from transformers import AutoModelForCausalLM
import torch
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3-8B",
    torch_dtype=torch.bfloat16,          # FA needs fp16/bf16
    attn_implementation="flash_attention_2",
)   # every attention layer now runs the tiled kernel

    Verify it's exact (do this once)
    
        
    Because Flash is exact, its output should match the naive path to within floating-point tolerance. A quick allclose is a great sanity check — and a reminder that you're trading zero accuracy for a huge memory/speed win.
    
        
    ref  = naive_attention(Q, K, V)
fast = F.scaled_dot_product_attention(Q, K, V)
assert torch.allclose(ref, fast, atol=1e-2)   # same answer, fp tolerance
print("max abs diff:", (ref - fast).abs().max().item())
